## 5.2 系统数据库设计

### 5.2.1 数据库整体架构

本系统采用"链上存证 + 链下存储"的混合架构。链上（Hyperledger Fabric）存储溯源关键摘要数据，包括批次哈希、操作哈希和时间戳，具有不可篡改性；链下（MySQL 8.0）存储完整的业务数据，包括用户信息、批次详情、溯源节点记录等结构化数据；图片、PDF 等非结构化文件通过 IPFS 分布式存储，并将内容哈希（CID）同步上链，确保文件防篡改。

**表5-3  数据库核心表结构说明**

| 表名 | 中文名 | 主要字段 | 说明 |
|------|--------|----------|------|
| `users` | 用户角色表 | id, username, password_hash, role, org_name | 支持5种角色，bcrypt加密 |
| `agri_batches` | 农产品批次表 | batch_no, trace_code, variety, quantity, origin_lat/lng | 一物一码，含地理坐标 |
| `trace_records` | 溯源节点记录表 | node_type, operator_id, env_data(JSON), tx_hash | 7种节点类型，JSON存环境数据 |
| `planting_records` | 种植记录表 | field_area, fertilizer_type, irrigation_type, soil_ph | 精细化种植管理 |
| `harvest_records` | 采收记录表 | harvest_method, actual_quantity, sugar_content, hardness | 采收质量数据 |
| `processing_records` | 加工记录表 | process_type, operator_id, operator_phone, pack_spec | 责任到人 |
| `inspection_records` | 质检记录表 | pesticide_result, heavy_metal_result, microbe_result | 三项检测结果 |
| `logistics_records` | 物流追踪表 | vehicle_no, driver_name, depart_lat/lng, arrive_lat/lng | 含地理坐标 |
| `certificates` | 电子合格证表 | cert_no, issue_org, qr_code_url, tx_hash | 含二维码和上链哈希 |
| `dashboard_stats` | 仪表盘统计表 | total_batches, total_visitors, total_queries, total_tx | 每日聚合统计 |
| `captchas` | 验证码表 | key, code, expired_at, used | 图形验证码管理 |

### 5.2.2 核心表关系设计

溯源数据模型以 `agri_batches`（批次表）为核心，向上关联 `users` 表中的种植户信息，向下通过 `batch_id` 外键关联种植、采收、加工、质检、物流、合格证等全链路记录表。`trace_records` 表作为通用溯源节点记录表，通过 `node_type` 字段区分不同环节，`env_data` 字段采用 JSON 类型存储各环节特有的环境参数，兼顾了数据结构的灵活性与查询效率。

## 5.3 系统功能展示

### 5.3.1 系统用例分析

本系统面向五类参与主体，分别为系统管理员、种植户、质检员、物流商和销售商，同时提供面向消费者的公开溯源查询入口。各角色的核心用例如图5-1所示。

![图5-1 溯源系统用例图](pic/图5-1  溯源系统用例图.jpg)

系统管理员负责用户账号的创建与权限管理、系统参数配置以及区块链节点监控；种植户负责录入种植信息、上传现场图片并发起采收申请；质检员负责录入检测结果并生成电子合格证；物流商负责记录运输节点信息和实时位置；销售商负责确认收货并更新上架状态；消费者通过扫描产品二维码访问公开溯源查询页面，获取完整的溯源链路信息。

### 5.3.2 用户登录与注册

系统登录页面采用角色选择 + 账号密码 + 图形验证码的三重验证机制，如图5-3所示。注册流程要求填写用户名、真实姓名、联系电话、所属机构、角色类型及图形验证码，提交后由系统管理员审核激活，确保溯源链条参与主体的真实性与可追溯性。

![图5-3 系统登录页面](pic/图5-3  系统登录页面.png)

前端登录请求通过 HTTPS 加密传输，后端采用 bcrypt 算法（cost=10）对密码进行哈希存储，JWT Token 有效期为 24 小时，Token 刷新机制保障长时间操作的会话连续性。

### 5.3.3 系统主界面与仪表盘大屏

登录成功后进入系统主界面，左侧为多级菜单侧边栏，顶部为面包屑导航与用户信息栏，中央为仪表盘大屏，如图5-4所示。仪表盘实时展示平台访客数、溯源查询次数、区块链交易数、批次总量等核心指标，并通过 ECharts 折线图呈现近30天的趋势变化，辅助管理人员掌握平台运营态势。

![图5-4 溯源系统工作界面](pic/图5-4  溯源系统工作界面.png)

### 5.3.4 种植环节管理

种植管理模块支持种植户录入作物类型、种苗品种与来源、施肥类型与日期、灌溉方式、土壤pH值、农药使用情况等精细化信息，并提供图片上传控件，支持上传现场实拍照片作为佐证。地块地理坐标通过地图定位弹窗自动获取，确保产地信息的精确性与可核验性。每次记录提交后，系统自动调用区块链客户端将关键摘要数据上链，返回交易哈希（TxHash）和区块高度供后续审计。

### 5.3.5 加工环节管理

加工管理模块精细化记录每一次加工操作，支持清洗、烘干、分级、包装四种加工类型的独立录入。每条加工记录强制绑定操作人ID与联系电话，实现责任到人。包装环节需填写包装规格（如5kg/箱）、包装材料、总箱数及执行标准编号，生成的包装记录与批次表关联，为后续电子合格证的生成提供数据支撑。

### 5.3.6 物流追踪与地图可视化

物流追踪模块记录产品从产地到销售终端的全程流转信息，包括运输方式、车牌号、司机姓名与电话、出发地与目的地坐标、温控要求及实时温度。系统集成高德地图 JavaScript API，以折线图形式在地图上可视化展示物流路线，并在关键节点标注时间戳，如图5-5所示。

![图5-5 区块链浏览器界面](pic/区块链浏览器界面.png)

电子合格证模块是本系统的核心特色功能之一。质检员完成检测后，系统自动聚合批次信息、生产主体信息、检测结果（农药残留、重金属、微生物三项），生成符合《NY/T 1761-2009 农产品产地证明规范》的电子合格证。合格证包含以下核心要素：

合格证以弹窗形式展示，支持导出 PDF，并在右下角展示政府部门防伪印章，增强证书的权威性与可信度。

### 5.3.8 区块链浏览器

区块链浏览器模块展示 Hyperledger Fabric 联盟链的底层运行状态，包括当前区块高度、最新区块哈希、前一区块哈希、区块内交易数量及时间戳。交易详情弹窗展示完整的交易 ID、链码名称、调用参数、背书节点列表及国密 SM3 哈希值，为监管部门提供可审计的区块链存证查询界面。

## 5.4 系统性能初步评估

### 5.4.1 测试方案设计

本节采用 Hyperledger Caliper v0.5.0 对系统核心链码进行性能基准测试，同时使用 Apache JMeter 对后端 RESTful API 进行压力测试。测试场景覆盖三类核心业务：溯源记录上链（写操作）、溯源记录查询（读操作）和电子合格证生成（复合操作）。并发用户数梯度设置为 10、20、50、100、200，每个梯度持续运行 60 秒，采集 TPS、平均延迟、P95延迟、P99延迟和成功率等关键指标。

测试环境配置为：Fabric 网络部署于 3 台 8 核 16 GB 虚拟机，MySQL 和 Redis 部署于独立的 4 核 8 GB 虚拟机，测试客户端与服务端通过千兆内网连接。

### 5.4.2 吞吐量与延迟测试结果

图5-8展示了三类业务场景下系统吞吐量（TPS）随并发数增加的变化趋势。

![图5-8 系统吞吐量（TPS）随并发数变化曲线](python/output/图5-8_TPS并发曲线.png)

由图5-8可以观察到以下规律：溯源记录查询（读操作）的 TPS 显著高于写操作，在100并发时达到1456.8笔/秒，这与 Fabric 的读写分离设计一致，读操作无需经过共识排序，直接查询 Peer 节点的世界状态数据库；溯源记录上链（写操作）的 TPS 在50并发时达到156.3笔/秒的较优值，超过100并发后增速明显放缓，在200并发时仅为213.4笔/秒，表明系统在高并发写入场景下受到 Raft 共识延迟的制约。

图5-9展示了系统响应延迟随并发数的变化情况。

![图5-9 系统响应延迟随并发数变化曲线](python/output/图5-9_延迟并发曲线.png)

在50并发以内，溯源上链操作的平均延迟维持在320ms以内，满足业务可接受范围（≤500ms）；当并发数提升至100时，平均延迟升至503ms，P99延迟达到1123ms，表明系统在高并发场景下存在一定的延迟抖动。对于农产品溯源场景而言，溯源数据的录入并非高频实时操作，50并发以内的性能表现完全满足实际业务需求。

### 5.4.3 国密算法性能分析

图5-10对比了国密算法（SM2、SM3、SM4）与国际主流算法（RSA、ECDSA、SHA-256、AES）在相同硬件环境下的性能表现。

![图5-10 国密算法与国际算法性能对比](python/output/图5-10_国密算法性能对比.png)

测试结果表明，SM2 签名耗时（0.76ms）优于 ECDSA-P256（0.89ms），约为 RSA-2048（8.92ms）的 1/12，在保持同等128位安全强度的前提下，SM2 展现出显著的性能优势。SM3 哈希耗时（0.038ms）较 SHA-256（0.045ms）快约15.6%，这与 SM3 算法针对国产硬件平台的优化设计有关。SM4 加密耗时（0.015ms）略高于 AES-256（0.012ms），差异在可接受范围内。综合来看，国密算法的引入对系统整体性能的影响极为有限，可以认为国密改造在性能层面是可行的。

### 5.4.4 系统资源占用分析

图5-11展示了溯源上链场景下系统 CPU 使用率与内存占用随并发数的变化趋势。

![图5-11 系统资源占用随并发数变化曲线](python/output/图5-11_系统资源占用.png)

在50并发时，CPU 使用率为67.4%，内存占用891MB，系统运行平稳；在100并发时，CPU 使用率升至84.2%，接近80%的警戒线；在200并发时，CPU 使用率达到93.6%，存在资源瓶颈风险。建议在生产部署时配置水平扩展策略，当 CPU 使用率持续超过75%时自动扩容 Peer 节点。

### 5.4.5 上链开销分析

图5-12展示了溯源上链操作各环节耗时的占比分析（基于50并发、平均总耗时320ms）。

![图5-12 溯源上链操作各环节耗时占比分析](python/output/图5-12_上链耗时占比.png)

Fabric 共识耗时占比最高（38.5%），这是联盟链架构的固有开销，由 Raft 排序服务的批次打包机制决定；国密 SM2 签名开销占22.3%，是国密改造引入的主要额外开销；IPFS 存储占15.6%，主要来自图片文件的分布式存储延迟；MySQL 写入和网络传输合计占14.9%，属于正常范围。

### 5.4.6 综合性能评估

图5-13通过雷达图对本系统与传统中心化溯源系统进行多维度综合对比。

![图5-13 本系统与传统溯源系统综合性能雷达图](python/output/图5-13_综合性能雷达图.png)

**表5-5  系统综合性能评估汇总**

| 评估维度 | 本系统（区块链+国密） | 传统中心化系统 | 优势说明 |
|----------|----------------------|----------------|----------|
| 吞吐量 | 中等（写156 TPS@50并发） | 高（>1000 TPS） | 联盟链共识开销不可避免 |
| 响应速度 | 良好（320ms@50并发） | 优秀（<50ms） | 可接受范围内 |
| 安全性 | 优秀（国密SM2/SM3/SM4） | 一般（依赖中心节点） | 国密算法，符合国家标准 |
| 可靠性 | 优秀（99.8%@50并发） | 良好（依赖单点） | 多节点冗余，无单点故障 |
| 可扩展性 | 良好（水平扩展Peer） | 一般（垂直扩展为主） | 联盟链天然支持横向扩展 |
| 数据完整性 | 优秀（链上不可篡改） | 一般（依赖数据库事务） | 区块链不可篡改特性 |

综合评估结论：本系统在吞吐量和响应速度方面略低于传统中心化系统，这是引入区块链共识机制的固有代价；但在安全性、数据完整性和可靠性方面具有显著优势，尤其是国密算法的全面应用使系统满足国家信息安全等级保护（等保2.0）的密码算法要求。对于农产品溯源这一对数据可信度要求极高、对实时性要求相对宽松的应用场景，本系统的性能表现是合理且适宜的。

## 5.6 本章小结

本章围绕阿克苏苹果区块链溯源系统的工程实现展开，系统性地阐述了开发环境配置、数据库设计、各功能模块的实现细节以及系统性能评估结果。

在系统实现层面，后端采用 Go 1.22 + Gin + GORM 构建高性能 RESTful API，通过 bcrypt 密码加密、JWT 鉴权和角色权限中间件保障接口安全；数据库设计遵循《GB/T 29373-2012 农产品追溯要求》，设计了11张核心业务表，涵盖种植、采收、加工、质检、物流、合格证的全链路数据模型；区块链层采用国密改造版 Hyperledger Fabric v2.2，以 SM2 替代 ECDSA、SM3 替代 SHA-256，实现符合国家标准的密码算法体系；前端采用 Vue 3 + TypeScript 构建功能完整的管理后台，集成 ECharts 数据可视化、高德地图物流追踪、QRCode 二维码生成等特色组件。

在性能评估层面，基准测试结果表明：系统在50并发场景下，溯源上链 TPS 达156.3笔/秒，平均延迟320ms，成功率99.8%，满足农产品溯源业务的实际需求；国密算法（SM2/SM3）的性能不劣于同等安全级别的国际算法，引入国密改造对系统整体性能的影响在可接受范围内。综合来看，本系统在功能完整性、数据安全性和系统可靠性方面达到了预期设计目标，为农产品溯源信息化建设提供了一套可落地的技术方案。
